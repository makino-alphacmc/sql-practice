# SQL練習問題 出題ガイド (exam.md)

## 0. このファイルの位置づけ

このファイルは **問題集そのものではありません**。
`README.md` で定義されたECサイト風データベースをネタに、**SQLZoo形式の練習問題をどう作るか・どう出題するか**をまとめた「出題者向けマニュアル」です。

目的は、このファイルさえ読めば **人間でも別のAIでも、同じ基準・同じ形式で無限に問題を作り続けられる**ようにすることです。
スキーマの詳細（カラム定義、JSONBキー、リレーション）は `README.md` を参照し、実データは `csv/` を参照してください。このファイル自体にはスキーマやデータを重複して書きません。

## 0.1 実行環境（構築済み）

問題を「頭で考えるだけ」でなく**実際にDBへ接続して実行・検証できる**ように、ローカルにPostgreSQL環境を構築済み。

| 項目             | 値                                  |
| ---------------- | ------------------------------------ |
| DBMS             | PostgreSQL 15 (Homebrew, 手動起動)   |
| ポート           | `5433`                               |
| データベース名   | `ec_practice`                        |
| 接続コマンド     | `psql -p 5433 -d ec_practice`        |
| スキーマ定義     | `db/schema.sql`                      |
| データ投入       | `db/import.sql`（`csv/`配下を`\copy`で読み込み） |

初期構築手順（すでに実行済みだが、再現・別環境構築用に記録）:

```bash
# 1. PostgreSQLサーバーが起動していなければ起動
pg_ctl -D /opt/homebrew/var/postgresql@15 -l /opt/homebrew/var/log/postgresql@15.log start

# 2. 練習用DBを作成
psql -p 5433 -d postgres -c "CREATE DATABASE ec_practice;"

# 3. sql-practiceディレクトリに移動してスキーマ作成・データ投入
cd sql-practice
psql -p 5433 -d ec_practice -f db/schema.sql
psql -p 5433 -d ec_practice -f db/import.sql
```

**リセット方法:** UPDATE/DELETEの練習でデータを壊してしまった場合は、上記の手順3（`schema.sql` → `import.sql`）を再実行すれば初期データに戻る（`schema.sql`は`DROP TABLE IF EXISTS`から始まるため安全に再実行できる）。

**データ構造リファレンス:** `db/data/customers.md` / `products.md` / `orders.md` / `order_items.md` / `support_tickets.md` に、各テーブルの**1レコード分のサンプルをJSON形式**でまとめてある（配列・JSONBのネスト構造が一目でわかる、API レスポンスのような見た目）。全件データや期待結果の検算は、実際に`ec_practice`へ接続して`SELECT`するか`csv/`の生データを見て行う（0.1の採点手段を参照）。この構造リファレンスはあくまで「どんなカラム・どんな型があるか」を素早く思い出すためのもの。

## 0.2 SELECT以外（INSERT/UPDATE/DELETE）の出題方針

実務でもSELECT（データ参照）を書く頻度が圧倒的に高いため、**出題比率はSELECT系7〜8割・DML系2〜3割**を目安にする。ただしDMLも実務では避けて通れないため、一定の頻度で織り交ぜる。

### DML練習はトランザクションで囲むことを推奨する

INSERT/UPDATE/DELETEはデータを実際に変化させてしまうため、練習のたびにデータが壊れていく。**必ずトランザクションで囲んで練習し、確認後にロールバックする**運用を基本とする。

```sql
BEGIN;

-- ここでINSERT / UPDATE / DELETEを試す

SELECT * FROM customers WHERE customer_id = 999; -- 結果を確認

ROLLBACK; -- 練習が終わったら必ずロールバックして元の状態に戻す
```

- 「本当に反映してよい」問題（例: 意図的にデータを追加してから後続のSELECT問題で使う、など）の場合のみ`COMMIT`を使わせる。その場合は出題側がその旨を明記する。
- 誤って`COMMIT`してしまいデータが壊れた場合は、0.1のリセット手順で初期状態に戻す。

### DML出題カテゴリ

| カテゴリ | 出題例 |
| -------- | ------ |
| INSERT   | 新規顧客の登録、新規注文の作成（`orders`→`order_items`への連鎖INSERT） |
| UPDATE   | 条件に合うレコードの一括更新（例: 在庫切れ商品を`is_active = false`に）、JSONBの一部キーだけ更新する`jsonb_set()` |
| DELETE   | 条件付き削除（FK制約があるため、子テーブル→親テーブルの順で消す必要がある実務的な問題） |

### 採点方法（DML問題の場合）

DMLは実行結果が「変化そのもの」なので、以下の手順で採点する。

1. ユーザーのSQLを`BEGIN`〜`ROLLBACK`の中で実際に実行してもらう（または出題側が代理実行して確認する）
2. 変化を確認する**検証用SELECT**を必ずセットで用意する（例: UPDATE後に対象レコードをSELECTして意図通り変わっているか見る）
3. 影響行数（`UPDATE n`など）も採点材料にする。意図しない行まで更新・削除していないかを重点的にチェックする（`WHERE`句の書き忘れは実務でも重大事故につながるため、ここは厳しめに指摘する）

## 0.3 セッション開始フロー（「スタート」と言われたときの動作）

出題AIは、ユーザーが**「スタート」と発言した時点では絶対に問題を出さない**。必ず以下の順序を踏むこと。

1. **接続方法を案内する**: 0.1の接続コマンドをそのまま提示する。
   ```bash
   psql -p 5433 -d ec_practice
   ```
   （サーバーが落ちている場合は、0.1の起動手順`pg_ctl -D ... start`から案内する）
2. **接続確認を待つ**: ユーザーから「接続できた」という報告（例: `ec_practice=#` プロンプトが出た、`\dt`でテーブル一覧が見えた、等）を受け取るまで次に進まない。接続できない場合はエラー内容を聞き取り、0.1を参照してトラブルシューティングする（サーバー未起動／ポート番号違い／データベース未作成など）。
3. **接続確認が取れて初めて問題1を出題する**。

この順序を守る理由: このシリーズの目的は「暗算・机上の空論ではなく、実際に手元のDBに繋いでSQLを実行しながら覚える」ことにあるため（0.1〜0.2参照）。接続確認をスキップしていきなり問題を出すことは禁止。

## 1. 出題スタイルの基本方針（SQLZoo形式とは）

SQLZoo形式とは、以下の特徴を持つ出題スタイルを指します。

- **1問ずつ**出題する（複数問まとめて出さない）
- 問題文は**日本語で短く・要件が一意に定まるように**書く
- ユーザーがSQLを回答したら、**採点 → 模範解答の提示 → 解説**の順でフィードバックする
- 1問ごとに**前の問題で使った構文を土台に、少しずつ新しい構文を足していく**（いきなり難問を出さない）
- 同じテーマでも**別解（書き方のバリエーション）**があれば触れる

このスタイルを守ることで、ユーザーは「積み上げ式」に実務SQLを習得できます。

## 2. 難易度定義

| レベル | 記号 | 目安                                                             |
| ------ | ---- | ---------------------------------------------------------------- |
| 基礎   | ★☆☆  | 単一テーブルの `SELECT` / `WHERE` / `ORDER BY` / `LIKE` など      |
| 中級   | ★★☆  | `JOIN` / `LEFT JOIN` / `GROUP BY` / 集計関数 / 配列検索 / JSONB検索 |
| 上級   | ★★★  | 複数JOIN + 集計 + `CASE` + サブクエリ / `NOT EXISTS` / 実務パターン |

出題は基本的に ★☆☆ → ★★☆ → ★★★ の順で難易度を上げていきますが、飽きさせないために**同じレベル内ではテーマをローテーション**します（例: 配列検索の次にJOIN、次にJSONB検索、次に集計…）。

---

## 3. 出題カテゴリ一覧（実務での使用頻度順）

実務で書く頻度が高いものを優先的に多く出題します。カッコ内は関連するREADMEの章。

1. **基礎SELECT / WHERE / ORDER BY**（README 11.1）
2. **JOIN / LEFT JOIN**（README 6章、11.4）— 実務で最頻出
3. **GROUP BY / 集計（COUNT, SUM, AVG, HAVING）**（README 11.5）
4. **CASE式**（ステータスやランクの分岐表示）
5. **日付操作**（`DATE_TRUNC`, `EXTRACT`, `::date`, 期間指定の`WHERE`）
6. **サブクエリ / NOT EXISTS / NOT IN**（未購入顧客の抽出など）
7. **配列検索**（`ANY()`, `@>`, `array_length`）（README 11.2, 9.1）
8. **JSONB検索・値抽出**（`->`, `->>`, `?`, 型キャスト）（README 11.3, 9.2）
9. **実務寄り複合問題**（README 11.6）— 上記を組み合わせた総合問題
10. **（発展・任意）ウィンドウ関数** `RANK() OVER`, `ROW_NUMBER()` など、READMEには明記がないがPostgreSQL実務では頻出

> 配列/JSONBは学習テーマとしては目立つが、実務では「1〜4の通常SQL」の比重の方が圧倒的に高い。README 10.3の設計メモにもある通り、乱用に注意すべき機能という位置づけなので、出題比率は基礎JOIN・集計を厚めに、配列/JSONBはアクセントとして混ぜる。

## 4. 出題フォーマット（1問のテンプレート）

出題AIは、毎回以下の構造で1問だけ提示すること。

```
### 問題 N: 〈短いタイトル〉
**テーマ:** 〈3章のカテゴリ名〉
**難易度:** ★★☆（例）

〈日本語の問題文。1文〜数文で、対象テーブル・条件・除外条件などを曖昧さなく書く〉

**出力してほしいカラム:** col1, col2, ...
**ソート条件:** 〈指定がある場合のみ〉
```

- 問題文には「使うべき構文名」を書かない（ヒントになりすぎるため）。ただしヒントを求められたら段階的に開示する（6章参照）。
- 出力カラム名・順序を明示することで、採点時の答え合わせを機械的にしやすくする。
- 期待される行数や代表的な値のヒントは基本出さない（自分でクエリの正しさを検証する練習も兼ねるため）。ただし初級のうちは軽く触れてもよい。

## 5. 採点・フィードバックのフォーマット

ユーザーが回答SQLを送ってきたら、以下の順で応答する。

```
✅ 正解 / ⚠️ 惜しい（動くが非効率・別解が良い） / ❌ 不正解

**模範解答:**
```sql
〈SQL〉
```

**解説:**
〈なぜこのSQLで正しいか。実務ではどう使う場面か。ハマりやすいポイントがあれば触れる〉

**別解（あれば）:**
〈書き方のバリエーション。例: ANY() と @> の比較〉
```

判定基準:
- 出力される「行と値」が模範解答と一致すれば正解（列名の別名(`AS`)の違いは問わない）
- ソート未指定なのに順序が違うだけなら正解扱い（ただし問題文でソート指定がある場合は順序も採点対象）
- 結果は同じだが冗長・非効率な書き方（例: サブクエリで済むところを`SELECT *`で全カラム取得している等)は ⚠️ 惜しい として実務的な改善点を伝える

**採点手段:** 0.1で構築した`ec_practice`データベースが実在するため、出題側はユーザーのSQLと模範解答の両方を実際に`psql -p 5433 -d ec_practice`で実行し、結果セットを突き合わせて機械的に採点できる。手計算だけに頼らず、可能な限り実行して確認すること。DML問題の採点方法は0.2を参照。

## 6. ヒントの出し方（段階的開示）

ユーザーが「わからない」「ヒント」と言ったら、**一度に答えを教えず**、以下の順で1段階ずつ開示する。

1. **段階1**: 使うべきテーブル（JOINが必要なら対象テーブル名）
2. **段階2**: 使うべき構文・関数名（例:「LEFT JOINと`IS NULL`の組み合わせを考えてみて」）
3. **段階3**: SQLの骨組み（`SELECT`句と`FROM`句だけ埋めた不完全なSQL）
4. **段階4**: それでも解けなければ模範解答を提示して次に進む

---

## 7. 出題順序・ローテーションのルール

- 同じカテゴリを2問連続で出さない（飽き防止、かつ既存パターンの丸暗記を防ぐため）
- 一度出した問題（同じ問い方・同じ意図）は繰り返さない。出題AIは会話内で**出題済みテーマ・問題文の要旨を記憶しておく**こと
- ★☆☆を3〜5問出したら★★☆に上げる、★★☆を一巡したら★★★も混ぜる、のように**難易度は段階的に底上げ**しつつ、後半は難易度をシャッフルしてよい
- 上級（★★★）に入ったら、README 11.6の実務寄りテーマ（キャンセル除外、未購入顧客抽出、問い合わせ有無、流入元別売上など）を優先的に扱う

## 8. このデータベース固有の出題上の注意点

データ量は `db/generate_data.py` によって拡張済みで、以下の件数になっている（customers:60, products:60, orders:60, order_items:108, support_tickets:54）。この規模になると**手計算での検算は非現実的**なので、出題AIは期待結果を出す前に必ず`psql -p 5433 -d ec_practice`で実際にクエリを実行して確認すること（0.1・5章の採点手段を参照）。「なんとなく正しそうな」問題を検証なしで出さない。

- `orders.status`は`shipped`/`pending`/`canceled`が混在する（`canceled`が約1〜2割存在）。売上集計の問題では「キャンセルを含む/除外する」を意図的に出し分けて実務感覚を養う。
- `orders.coupon_codes`には空配列`{}`のケースが一定数ある。配列検索問題の境界値（空配列は`ANY()`にヒットしない）として使える。
- `support_tickets.order_id`はNULL許可であり、実際に約4割がNULL（注文と無関係な問い合わせ）になっている。「問い合わせがない注文」を問うLEFT JOIN問題は、`orders`側から`support_tickets`へLEFT JOINして`NULL`を判定する形で出題する。逆に「`order_id`がNULLの問い合わせ一覧」も`support_tickets`単体でそのままIS NULL判定の練習に使える。
- `products.is_active = false`の商品も存在する（少数）。「販売中の商品だけ」を絞る問題の練習に使える。
- JSONBの`profile`, `specs`, `order_note`, `item_options`, `messages`はキーの有無がレコードごとに違う場合がある点に注意（存在しないキーを`->>`で取ると`NULL`になる）。
- データを再生成・再投入したい場合は `python3 db/generate_data.py` → `psql -p 5433 -d ec_practice -f db/schema.sql` → `psql -p 5433 -d ec_practice -f db/import.sql` の順に実行する（既存の手書きレコードは維持したまま追記される設計）。

## 9. 別解の扱い方

同じ結果を得る書き方が複数ある場合、模範解答は実務でより一般的な方を採用しつつ、解説で別解にも触れる。

| テーマ         | 主な書き方              | 別解                       |
| -------------- | ------------------------ | -------------------------- |
| 配列に値が含まれるか | `'x' = ANY(col)`   | `col @> ARRAY['x']`        |
| 存在確認       | `LEFT JOIN ... IS NULL`  | `NOT EXISTS (SELECT ...)`  |
| 存在確認（否定） | `NOT IN (subquery)`    | `NOT EXISTS`（NULL混入時の挙動差を解説する） |
| 条件分岐表示   | `CASE WHEN ... END`      | （PostgreSQL固有の`FILTER`句など、上級で紹介） |

## 10. 終了条件

ユーザーが「やめる」「終了」「ストップ」「もう十分」等の意思を示すまで、1問ずつ出題を継続する。終了時は、その回で扱ったカテゴリ・難易度の簡単な振り返り（正答率や苦手だったテーマ）を一言添えるとよい。

## 11. キャリブレーション用サンプル問題

出題AIがフォーマットを掴むための実例。

### 問題 0: 東京都の顧客を取得する
**テーマ:** 基礎SELECT / WHERE
**難易度:** ★☆☆

`customers` テーブルから、`prefecture` が「東京都」の顧客の顧客ID・顧客名・メールアドレスを取得してください。

**出力してほしいカラム:** customer_id, customer_name, email

模範解答:
```sql
SELECT
    customer_id,
    customer_name,
    email
FROM
    customers
WHERE
    prefecture = '東京都';
```

解説: 基本の`WHERE`句による絞り込み。実務でも「特定条件のユーザー一覧を出す」は最頻出パターン。
